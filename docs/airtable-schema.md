# Airtable schema — Base "Hyrox Coach"

Seven tables. The athlete is a single user in this MVP, so most tables don't need a `user_id` column — but they're designed so adding one (for multi-user support later) is a one-column change without breaking anything.

## 1. `athlete_profile` — 1 row (the athlete)

Static and semi-static facts about the athlete. The agent reads this on every run to personalize tone, intensity, and recommendations.

| Field | Type | Example | Notes |
|---|---|---|---|
| `name` | Single line text | "Thibaud" | Display name |
| `birth_date` | Date | 1990-XX-XX | For age-based zones |
| `height_cm` | Number | 180 | |
| `weight_target_kg` | Number | 78 | Goal weight for race day |
| `hyrox_target_date` | Date | 2026-08-15 | Used to compute weeks-to-race and current phase |
| `hyrox_target_division` | Single select | "Open Men" / "Pro Men" / "Doubles" | |
| `coach_tone` | Single select | "supportive" / "drill-sergeant" / "neutral" | Editable via chat |
| `coach_severity` | Single select | "easy" / "balanced" / "hard" | Editable via chat |
| `equipment_home` | Multi-select | "kettlebell 16kg, dumbbells 10kg, pull-up bar" | Drives what workouts can be done at home |
| `equipment_kitchen` | Multi-select | "blender, oven, scale" | Drives meal feasibility |
| `dietary_preferences` | Multi-select | "omnivore" (default) — extensible to vegetarian/vegan/keto post-MVP | |
| `injuries_active` | Long text | "right shoulder rehab" | Modifies workout selection |
| `notes` | Long text | Free-form context the athlete wants the agent to remember | |

## 2. `training_plan` — multiple rows (one per planned session)

The structured training plan for the current Hyrox prep cycle. Generated by the agent on demand (e.g. "build me week 5") and editable both via chat and directly in Airtable.

| Field | Type | Example | Notes |
|---|---|---|---|
| `date` | Date | 2026-05-19 | Primary key for "the session of this day" |
| `phase` | Single select | "Base" / "Build" / "Peak" / "Taper" | Auto-computed from `weeks_to_race` |
| `week_number` | Number | 4 | Week of the prep |
| `session_type` | Single select | "Hyrox simulation" / "Strength" / "Z2 endurance" / "Intervals" / "Rest" / "Mobility" | |
| `description` | Long text | "5 rounds: 1km row + 50 wall balls" | What the athlete actually does |
| `duration_min` | Number | 60 | Planned duration |
| `intensity` | Single select | "easy" / "moderate" / "hard" / "max" | |
| `equipment_required` | Multi-select | (matches `equipment_home` options) | |
| `notes` | Long text | "Focus on pacing on the row" | Coaching cues |
| `status` | Single select | "planned" / "done" / "skipped" / "modified" | Updated by Workflow D after Strava sync |

## 3. `daily_logs` — one row per completed activity

What actually happened. Populated automatically by Workflow D (Strava sync) and manually editable from chat. The agent compares `daily_logs` against `training_plan` to gauge adherence and adjust future sessions.

| Field | Type | Example | Notes |
|---|---|---|---|
| `date` | Date | 2026-05-19 | |
| `activity_type` | Single line text | "Running" / "WeightTraining" / "Rowing" | Strava field name |
| `duration_min` | Number | 58 | |
| `distance_km` | Number | 9.2 | If applicable |
| `avg_heart_rate` | Number | 152 | |
| `perceived_effort` | Single select | "easy" / "moderate" / "hard" / "max" | Set via chat after the session |
| `strava_activity_id` | Single line text | "12345678901" | Idempotency key for Workflow D |
| `notes` | Long text | "Legs felt heavy" | |

## 4. `nutrition_logs` — one row per logged meal (optional in MVP)

In Option A (MVP), this table is mostly empty — the agent suggests meals without tracking them. The schema exists so the Day-5 stretch goal (meal photo → macros) can write here.

| Field | Type | Example | Notes |
|---|---|---|---|
| `date` | Date | 2026-05-19 | |
| `meal_type` | Single select | "breakfast" / "lunch" / "dinner" / "snack" | |
| `description` | Long text | "Oats + banana + whey" | Free-form |
| `estimated_kcal` | Number | 550 | From Vision LLM when meal photos arrive |
| `estimated_protein_g` | Number | 35 | |
| `estimated_carbs_g` | Number | 80 | |
| `estimated_fat_g` | Number | 12 | |
| `source` | Single select | "manual" / "photo" / "suggested" | How the row was created |

## 5. `body_metrics` — one row per Withings screenshot ingested

Populated by Workflow C (screenshot ingestion).

| Field | Type | Example | Notes |
|---|---|---|---|
| `date` | Date | 2026-05-19 | Date the measurement was taken (from the screenshot) |
| `weight_kg` | Number | 79.4 | |
| `body_fat_pct` | Number | 14.2 | If shown in screenshot |
| `muscle_mass_kg` | Number | 38.1 | If shown |
| `sleep_hours` | Number | 7.2 | If from a sleep-tracker screenshot |
| `resting_hr` | Number | 52 | If shown |
| `screenshot_source` | Single select | "withings" / "strava" / "other" | |
| `screenshot_url` | URL | Telegram file URL or Airtable attachment | Audit trail |

## 6. `schedule_events` — one row per Google Calendar event for the next 7 days

Pulled by Workflow A each morning. Used by the agent to know whether the athlete will be travelling, in long meetings, etc. This is **read-only from the agent's perspective**: Workflow A wipes and refills the table each morning.

| Field | Type | Example | Notes |
|---|---|---|---|
| `date` | Date | 2026-05-19 | |
| `start_time` | Single line text | "09:00" | Local time |
| `end_time` | Single line text | "10:30" | |
| `title` | Single line text | "Client meeting" | |
| `location` | Single line text | "Berlin" or "Home" | Used to suggest travel-friendly workouts |
| `is_all_day` | Checkbox | false | |
| `calendar_id` | Single line text | "primary" | If multi-calendar support added later |

## 7. `chat_memory` — rolling buffer of recent chat turns

Used by Workflow B (conversational chat) to maintain context across days. Older rows are pruned after 30 days to keep size manageable.

| Field | Type | Example | Notes |
|---|---|---|---|
| `timestamp` | Date with time | 2026-05-19T14:23:00 | |
| `role` | Single select | "user" / "assistant" | |
| `content` | Long text | "I'm feeling sore today, can we swap the run for mobility?" | |
| `session_id` | Single line text | "tg_chat_6686274023" | Same Telegram chat ID convention as the Lead Capture lab |

## Cross-table relationships

There are no formal Airtable "linked records" in the MVP (to keep things simple). Joins are done in n8n / Streamlit by `date`:

```
training_plan.date == daily_logs.date == nutrition_logs.date == 
body_metrics.date == schedule_events.date
```

This denormalised structure is intentional for an MVP. If multi-user support is added, all tables get a `user_id` column and joins use `(user_id, date)`.
