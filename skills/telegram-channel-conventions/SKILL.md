# Skill: Telegram channel conventions

## When to use

Any node that sends a message to Telegram (athlete brief, chat reply, system error notification).

## Channels

This project uses **one bot**, but two logical "channels":

| Channel | Purpose | Chat ID stored as |
|---|---|---|
| Athlete | All coaching content: morning briefs, chat replies, screenshot confirmations | `TELEGRAM_ATHLETE_CHAT_ID` (it's the athlete's personal chat with the bot) |
| System | Error notifications, deploy notices, anything technical | `TELEGRAM_SYSTEM_CHAT_ID` (a private channel the athlete created, just for system events) |

Don't cross-post. The athlete should never see a workflow error in the same thread as their morning brief.

## Formatting

- Use Telegram's MarkdownV2 dialect, **not** standard markdown. Escape special characters: `_`, `*`, `[`, `]`, `(`, `)`, `~`, `` ` ``, `>`, `#`, `+`, `-`, `=`, `|`, `{`, `}`, `.`, `!`.
- Emojis are encouraged for the athlete channel (☀️ for briefs, 💪 for encouragement, 📚 for sources). Use them at the start of section headers, not mid-sentence.
- Keep messages under 1500 characters when possible. Telegram's hard limit is 4096; long messages get rendered poorly on mobile.

## The athlete channel — morning brief format

The structured brief format is defined in `docs/project-specification.md`. Don't reinvent it per workflow. The brief is generated by the AI Agent node and sent verbatim — no extra wrapping.

## The athlete channel — chat replies

Replies are conversational. Always under 200 words. End with a question or call to action when appropriate.

If the agent has modified Airtable as part of the reply (e.g. swapped a session), explicitly mention it:
> "✓ I've updated tomorrow's session to a 30-min mobility flow. Anything else?"

## The athlete channel — screenshot confirmations

When Workflow C extracts data from a screenshot, always echo what was extracted before saving:

> "📸 I see: weight 79.4 kg, body fat 14.2%, muscle mass 38.1 kg, taken May 19. Saving to body metrics — correct?"

For the MVP, save immediately without waiting for confirmation, but include the values so the athlete can correct via chat if needed.

## The system channel — error notifications

Format:

```
🚨 Workflow Failed: <workflow name>
Node: <node name where failure occurred>
Error: <error message>
Time: <ISO timestamp>
Execution: <link to n8n execution page>
```

Keep these terse. The athlete will read them on their phone while doing something else.

## Anti-patterns

- ❌ Don't send images from the agent. Telegram bots can but it adds complexity and our use case doesn't need it.
- ❌ Don't send "typing..." indicators or fake delays. Either reply immediately when ready, or use the n8n Telegram node's built-in deferral.
- ❌ Don't use `parse_mode: HTML`. Stick to MarkdownV2 everywhere for consistency.
- ❌ Don't send the same message twice. If a workflow needs to confirm an action it already took, edit the previous message instead (Telegram's `editMessageText` API).
